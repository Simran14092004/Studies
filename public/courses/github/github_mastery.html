Version Control & Collaboration

Git is the most widely used distributed version control system in the world. GitHub wraps Git with collaboration features — pull requests, issues, Actions, and more. As an AIML developer, mastering Git is non-negotiable.

Why Git? Every professional software project uses version control. Git lets you track changes, collaborate safely, revert mistakes, and ship confidently.

Installation & Configuration

# Install on Ubuntu/Debian
sudo apt install git

# Verify
git --version

# FIRST THING: Configure your identity
git config --global user.name "Simran Singh"
git config --global user.email "your@email.com"
git config --global core.editor "code --wait"   # VSCode as editor
git config --list   # See all config

SSH Key Setup (recommended)

# Generate SSH key
ssh-keygen -t ed25519 -C "your@email.com"

# Copy public key
cat ~/.ssh/id_ed25519.pub
# Paste this in GitHub → Settings → SSH Keys

# Test connection
ssh -T git@github.com

Core Git Commands

Initialize

git init — create a new repo in current dir

git clone <url> — copy existing repo

Staging & Committing

git add . — stage all changes

git commit -m "message" — snapshot

Status & Logs

git status — what's changed

git log --oneline --graph — visual history

Undoing Changes

git restore file — discard working changes

git reset HEAD~1 — undo last commit (keep changes)
# Complete workflow
git init
echo "# My Project" > README.md
git add README.md
git commit -m "feat: initial commit"

# See what changed
git diff
git diff --staged   # after git add

# Amend last commit message
git commit --amend -m "corrected message"

Branching Strategy

Branches let you develop features in isolation without breaking main/production code.

# Create and switch to new branch
git checkout -b feature/user-auth
# or modern syntax:
git switch -c feature/user-auth

# List branches
git branch        # local
git branch -a     # all (including remote)

# Switch branches
git switch main

# Delete branch (after merging)
git branch -d feature/user-auth

# Rename branch
git branch -m old-name new-name
Naming Convention: Use prefixes like feature/, fix/, hotfix/, chore/, docs/ for clear communication in team projects.

Remote Repositories

# Add remote origin
git remote add origin git@github.com:username/repo.git

# View remotes
git remote -v

# Push branch to GitHub
git push -u origin main     # -u sets upstream tracking
git push                    # subsequent pushes

# Pull latest changes
git pull                    # fetch + merge
git fetch                   # only download, don't merge

# Push new branch
git push origin feature/new-feature

Fork & Contribute to Open Source

# 1. Fork on GitHub (click Fork button)
# 2. Clone YOUR fork
git clone git@github.com:yourname/project.git
# 3. Add upstream remote
git remote add upstream git@github.com:original/project.git
# 4. Keep fork updated
git fetch upstream
git merge upstream/main

Merge & Rebase

# Merge feature into main (creates merge commit)
git switch main
git merge feature/user-auth

# Rebase (replays commits on top of main — cleaner history)
git switch feature/user-auth
git rebase main

# Squash merge (combines all feature commits into one)
git merge --squash feature/user-auth
git commit -m "feat: add user authentication"

# Interactive rebase (edit/squash/reorder last N commits)
git rebase -i HEAD~3
Golden Rule: Never rebase shared/public branches (main, develop). Only rebase your private feature branches before merging.

Resolving Merge Conflicts

# When you see:
<<<<<<< HEAD
your version of code
=======
incoming version of code
>>>>>>> feature/branch

# Steps:
# 1. Open the file, decide which version (or combine both)
# 2. Remove conflict markers <<<< ==== >>>>
# 3. Stage resolved file
git add conflicted-file.js
# 4. Complete the merge
git commit
Pro tip: Use git mergetool or VSCode's built-in merge editor for visual conflict resolution.

GitHub Actions (CI/CD)

Automate your workflows: run tests, deploy apps, lint code — triggered on push/PR/schedule.

# .github/workflows/ci.yml
name: CI Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: pip install -r requirements.txt
      - run: pytest tests/

  deploy:
    needs: test
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v4
      - run: echo "Deploy to Vercel/Heroku here"

Stash, Tags & Releases

# STASH — temporarily save dirty work
git stash                    # save current changes
git stash pop                # restore latest stash
git stash list               # see all stashes
git stash apply stash@{1}    # apply specific stash

# TAGS — mark release points
git tag v1.0.0                          # lightweight tag
git tag -a v1.0.0 -m "First release"   # annotated tag
git push origin v1.0.0                 # push tag to GitHub
git push origin --tags                 # push all tags

# Checkout specific tag
git checkout v1.0.0

Common Interview Questions

Q: Difference between git fetch and git pull?
A: git fetch downloads remote changes but doesn't modify your working directory. git pull = git fetch + git merge. Fetch is safer to inspect changes first.
Q: What is HEAD in Git?
A: HEAD is a pointer to the current commit you're working on. It normally points to the tip of the current branch. git log --oneline shows HEAD position.
Q: git reset vs git revert?
A: reset rewrites history (dangerous on shared branches). revert creates a new commit that undoes changes — safe for public branches.
Q: What is a detached HEAD?
A: When HEAD points directly to a commit instead of a branch. Happens when you git checkout <commit-hash>. Create a new branch to save work done in this state.
Q: Explain git cherry-pick
A: Applies a specific commit from one branch to another. Useful for applying a hotfix to multiple branches: git cherry-pick <commit-hash>
