CHAPTER · 01
Introduction to DBMS

A Database Management System (DBMS) is an organized collection of interrelated data together with a set of programs to access, define, and manipulate it. It hides the gory details of how bytes are laid out on disk and presents a clean, logical view to the application.

1.1 File System vs DBMS

AspectFile SystemDBMS
Data RedundancyHighControlled
ConsistencyHard to maintainEnforced via constraints
Concurrent AccessApplication-level locksBuilt-in transactions
RecoveryManual / backupsACID + log-based recovery
SecurityOS file permissionsFine-grained GRANT/REVOKE
QueryingCustom codeDeclarative SQL

1.2 Three-Schema Architecture (ANSI/SPARC)

   ┌────────────────────────────────┐
   │   External  View 1    View 2   │   ← User views (logical subset)
   └────┬────────────┬──────────────┘
        │  Logical Data Independence
   ┌────▼────────────▼──────────────┐
   │     Conceptual / Logical       │   ← Entire DB schema (tables, FKs)
   └────┬───────────────────────────┘
        │  Physical Data Independence
   ┌────▼───────────────────────────┐
   │ Internal / Physical Schema     │   ← Indexes, pages, file layout
   └────────────────────────────────┘
Data Independence is the ability to change one schema level without affecting the level above it. Physical independence is easier to achieve; logical independence is harder because views ride on the logical schema.

1.3 DBMS Architecture Types

  • 1-tier: User directly sits on the DBMS (e.g., DBA running scripts).
  • 2-tier: Client ↔ Server (classic ODBC/JDBC apps).
  • 3-tier: Client ↔ Application server ↔ DBMS (almost every web app).

1.4 Data Models — A Family Tree

▸ Hierarchical, Network, Relational, Object-Oriented, NoSQL
  • Hierarchical (IMS, 1966): Tree-shaped; one parent per child. Fast traversal, painful queries.
  • Network (CODASYL): Graph; child may have many parents. More flexible but complex.
  • Relational (Codd, 1970): Tables (relations). Declarative SQL. The GATE star.
  • Object-Oriented: Class hierarchies (e.g., db4o).
  • NoSQL / Document / Key-Value / Column-family / Graph: Trade strict schema/ACID for horizontal scale.

1.5 Database Users & the DBA

  • Naïve users — fill forms, click buttons.
  • Application programmers — write business logic.
  • Sophisticated users — analysts firing raw SQL.
  • DBA — schema design, performance tuning, access control, backups.

1.6 Numerical: Storage Estimation

storage_required = rows × (Σ column_sizes + row_overhead) × (1 + index_overhead)
Worked Example. A table Student(roll INT(4B), name CHAR(20), cgpa FLOAT(4B)) with 1,00,000 rows and 20% index overhead.
Row size = 4 + 20 + 4 = 28 B → Data = 28 × 105 = 2.8 MB → With indexes ≈ 3.36 MB.
Did you know? The term “database” first appeared in print in a 1962 System Development Corporation technical memo — long before SQL was even imagined.

CHAPTER · 02
Data Modeling with the ER Model

The Entity-Relationship (ER) model is a graphical language for capturing the real-world meaning of data before mapping it to tables.

2.1 Core Vocabulary

  • Entity — a real-world object (a student, an account). Strong entity has its own PK; weak entity needs an owner.
  • Attribute — property of an entity. Types: simple, composite, derived, multi-valued, key.
  • Relationship — association between entity sets. Degree = number of entity sets involved.
  • Cardinality — 1:1, 1:N, M:N.
  • Participation — total (double line) or partial (single line).

2.2 ER Diagram Symbols

  ┌──────────┐    ◇  Relationship      ⬭  Attribute
  │  Entity  │    ◆  Identifying rel    ⬭⬭ Multi-valued
  └──────────┘    ║   Weak entity       (⬭) Derived
                  •—  Partial         •═  Total participation

2.3 Example — University Mini-Schema

  STUDENT ────── enrolls ──────M:N────── COURSE
   (rollNo*)                              (cid*)
      │
      │ 1
      │
   advisor (1:N)
      │
      │ N
      ▼
  FACULTY (fid*) ──── teaches (1:N) ──── COURSE
Total participation: every STUDENT has an advisor → double line from STUDENT to advisor.

2.4 EER — Specialization & Generalization

  • Specialization (top-down): split a superclass into subclasses (Person → Student, Faculty).
  • Generalization (bottom-up): combine subclasses into a superclass.
  • Aggregation: treat a relationship as a higher-level entity (often needed when a relationship participates in another relationship).
  • Disjoint vs. Overlapping; Total vs. Partial.

2.5 ER → Relational Mapping Rules

▸ The 7 rules every GATE aspirant must memorize
  1. Strong entity → table with all simple attributes; PK = entity key.
  2. Weak entity → table that includes the owner's PK as part of its composite key.
  3. 1:1 relationship → merge into one of the entity tables (prefer total-participation side) using an FK with UNIQUE.
  4. 1:N → put FK on the N-side referencing the 1-side.
  5. M:N → create a new table with FKs to both, PK = (FK1, FK2).
  6. Multi-valued attribute → new table (entity_PK, value).
  7. Specialization → either (a) one table for superclass + tables for subclasses, (b) one table per subclass with inherited columns, or (c) single table with discriminator. Choice depends on disjointness/totality.

2.6 Numerical — Counting Tables

Problem. An ER diagram has 3 strong entities, 1 weak entity, 2 M:N relationships, and 1 multi-valued attribute on a strong entity. How many relations are produced?
Solution. 3 (strong) + 1 (weak) + 2 (M:N) + 1 (MVA) = 7 tables. Note: 1:N relationships do NOT produce extra tables.
Pitfall: Aggregation is needed when a relationship has its own attributes and participates in another relationship. Don't confuse aggregation with composition (UML).
Did you know? Peter Chen’s original 1976 ER paper is one of the top-10 most cited computer science papers ever — over 10,000 citations.

CHAPTER · 03
Relational Data Model & Languages

Codd's relational model treats data as relations (sets of tuples). Two complementary query languages exist: Relational Algebra (procedural) and Relational Calculus (declarative).

3.1 Terminology

FormalInformal
RelationTable
TupleRow
AttributeColumn
DomainData type / set of allowed values
DegreeNo. of columns
CardinalityNo. of rows

3.2 Keys

  • Candidate Key: minimal super key.
  • Primary Key: chosen candidate key.
  • Super Key: any set that uniquely identifies tuples (need not be minimal).
  • Alternate Keys: candidates not chosen as primary.
  • Foreign Key: attribute(s) referencing PK of another relation.
  • Composite Key: candidate key with ≥ 2 attributes.

3.3 Integrity Constraints

  • Domain — values must come from the declared domain.
  • Entity Integrity — no part of a PK may be NULL.
  • Referential Integrity — FK value must match some PK value or be NULL (if allowed).
  • Key constraint — UNIQUE.

3.4 Relational Algebra — Operators

σ (selection) · π (projection) · ρ (rename) ∪ (union) · − (set diff) · ∩ (intersection) · × (Cartesian) ⋈θ (theta-join) · ⋈ (natural join) · ⟕ ⟖ ⟗ (outer joins) · ÷ (division) 
σ_{age > 20} (Student)        — picks rows
π_{name, cgpa} (Student)      — picks columns
Student ⋈ Enroll              — joins on common attrs
Student ⋈_{rollNo = sid} Enroll — theta join
Course ÷ Required             — "find courses that satisfy ALL"

3.5 Relational Calculus

  • Tuple Relational Calculus (TRC): { t | P(t) }, e.g. { s | s ∈ Student ∧ s.cgpa > 8 }
  • Domain Relational Calculus (DRC): { ⟨x1,…,xn⟩ | P(x1,…,xn) }
RA, safe TRC, safe DRC, and SQL (minus aggregates & nulls) are equivalent in expressive power.

3.6 Division — The Tricky One

Given R(A,B) and S(B), R ÷ S returns all A-values that pair with every B in S. Classic use: "students who have taken all required courses".

Identity: R ÷ S = πA(R) − πA( ( πA(R) × S ) − R )

3.7 Numerical — Tuple Counts

Problem. R has 200 tuples, S has 100 tuples on a common attribute. What's the max/min cardinality of R ⋈ S (natural join)?
Solution. Min = 0 (if no values match). Max = 200 × 100 = 20,000 (if the common attribute has the same value in every row of both — extremely skewed, but the algebraic upper bound).
Common trap. Set operators (∪, −, ∩) require union compatibility: same arity and corresponding domains. Otherwise: type error.
Did you know? Codd's 1970 paper “A Relational Model of Data for Large Shared Data Banks” was nearly rejected internally at IBM — they thought it was “too theoretical to ever be efficient”.

CHAPTER · 04
SQL & PL/SQL

SQL is the lingua franca of relational databases. It has three sub-languages: DDL (define), DML (manipulate), DCL (control). PL/SQL is Oracle's procedural extension.

4.1 The SELECT Anatomy

SELECT   col_list / aggregates
FROM     table_refs (with JOINs)
WHERE    row-level predicates
GROUP BY grouping cols
HAVING   group-level predicates
ORDER BY sort cols
LIMIT    n;
Pitfall: WHERE filters rows before grouping; HAVING filters after. Aggregates cannot appear in WHERE.

4.2 Joins — All Five

  • INNER JOIN — matched pairs.
  • LEFT OUTER JOIN — keep all left, NULL the unmatched right.
  • RIGHT OUTER JOIN — mirror.
  • FULL OUTER JOIN — both sides padded.
  • CROSS JOIN — Cartesian product.
  • Self-join — table joined with an alias of itself (e.g., employee–manager).

4.3 Aggregates & NULL semantics

  • COUNT(*) counts rows including NULLs; COUNT(col) ignores NULLs.
  • NULL is unknown: NULL = NULL → unknown; use IS NULL.
  • Three-valued logic: TRUE / FALSE / UNKNOWN.

4.4 Subqueries & Set Operations

  • Scalar: returns single value.
  • Row: returns single row.
  • Correlated: inner query references outer columns; evaluated per outer row.
  • Set ops: UNION, UNION ALL, INTERSECT, EXCEPT/MINUS.

4.5 Views

CREATE VIEW TopStudents AS
SELECT rollNo, name, cgpa
FROM   Student
WHERE  cgpa >= 9.0;
A view is updatable only if it's defined on a single table without aggregates, DISTINCT, GROUP BY, or set ops, and includes the table's PK.

4.6 Indexes

  • Primary — on PK; data is sorted by it (clustered).
  • Secondary — non-clustered; extra lookup level.
  • Dense vs Sparse: dense has one entry per record; sparse has one per data block.
  • B+ Tree — default; supports range queries.
  • Hash — O(1) equality, no range.

4.7 Transactions in SQL

BEGIN;
  UPDATE Account SET bal = bal - 1000 WHERE id = 1;
  UPDATE Account SET bal = bal + 1000 WHERE id = 2;
COMMIT;   -- or ROLLBACK on failure

4.8 PL/SQL — Quick Tour

DECLARE
   v_avg NUMBER;
BEGIN
   SELECT AVG(salary) INTO v_avg FROM Employee;
   IF v_avg < 50000 THEN
      DBMS_OUTPUT.PUT_LINE('Underpaid team');
   END IF;
END;
  • Cursors: implicit (single row) or explicit (multi-row).
  • Triggers: BEFORE / AFTER on INSERT/UPDATE/DELETE.
  • Stored Procedures & Functions: server-side modules.
  • Exceptions: NO_DATA_FOUND, TOO_MANY_ROWS, user-defined.

4.9 Numerical — Query Output Counting

Problem. Table Emp(eid, did, sal) has 5 rows: (1,A,100), (2,A,200), (3,B,300), (4,B,400), (5,C,500). What does SELECT did, AVG(sal) FROM Emp GROUP BY did HAVING AVG(sal) > 200; return?
Solution. Group averages: A→150, B→350, C→500. HAVING filters out A. Output: 2 rows — (B, 350), (C, 500).
Did you know? SQL was originally called SEQUEL (Structured English Query Language). It was renamed to SQL due to a trademark conflict — and IBM developed it at the San Jose Research Lab in the early 1970s.

CHAPTER · 05
Database Design & Normalization

Normalization eliminates insertion, update, and deletion anomalies by progressively decomposing relations based on Functional Dependencies (FDs).

5.1 Functional Dependencies

Armstrong's Axioms: Reflexivity: if Y ⊆ X then X → Y Augmentation: if X → Y then XZ → YZ Transitivity: if X → Y and Y → Z then X → Z Derived rules: Union, Decomposition, Pseudotransitivity

5.2 Attribute Closure (X⁺)

Algorithm.
  1. Start with X⁺ = X.
  2. Repeat: for each FD A → B with A ⊆ X⁺, set X⁺ := X⁺ ∪ B.
  3. Stop when no change.
X is a candidate key iff X⁺ = R and no proper subset has full closure.

5.3 The Normal Forms

FormDefinitionAnomaly addressed
1NFAll attributes atomicMulti-valued attributes
2NF1NF + no partial dependency of non-prime on a candidate keyPartial key redundancy
3NF2NF + for every FD X→A: either X is super key OR A is primeTransitive dependencies
BCNFFor every non-trivial FD X→Y, X is a super keyAll FD-based anomalies
4NFBCNF + no non-trivial multi-valued dependency (MVD)Independent multi-values
5NF (PJNF)4NF + every join dependency is implied by candidate keysSpurious tuples from joins

5.4 BCNF vs 3NF

  • 3NF allows X → A even when X is not super key, as long as A is prime.
  • BCNF doesn't allow that.
  • 3NF is always achievable as a lossless + dependency-preserving decomposition. BCNF is lossless but may lose dependency preservation.

5.5 Decomposition Properties

  • Lossless Join: R = R1 ⋈ R2; required when splitting. R1 ∩ R2 must be a super key of at least one of R1 or R2.
  • Dependency Preservation: closure of union of projected FDs equals closure of original F.

5.6 Numerical — Closure & Keys

Problem. R(A,B,C,D,E) with F = { A → BC, CD → E, B → D, E → A }. Find all candidate keys.
Solution sketch. Closures: A⁺ = {A,B,C,D,E} ✓; B⁺ = {B,D} ✗; D⁺ = {D}; E⁺ = {A,B,C,D,E} ✓; CD⁺ = {A,B,C,D,E} ✓; BC⁺ = {A,B,C,D,E} ✓. Minimality: A, E are candidate keys. CD and BC are also candidate keys (no proper subset closes to R).
Candidate keys: {A}, {E}, {BC}, {CD}.

5.7 Numerical — Decomposing into BCNF

Problem. R(A,B,C) with FDs {A → B, B → C}. Is R in BCNF? If not, decompose.
Solution. Candidate key = {A}. B → C violates BCNF (B not super key). Decompose: R1(B, C) [FD B→C, key B] and R2(A, B) [FD A→B, key A]. Lossless: R1 ∩ R2 = {B} which is the key of R1 ✓. Dependency preservation: A→B in R2; B→C in R1 ✓. Both BCNF.
Trap: Sometimes a relation in 3NF is not in BCNF; classic example: R(A,B,C) with FDs {AB → C, C → B}. Candidate keys: AB, AC. C→B violates BCNF (C not super key) but is allowed in 3NF because B is prime.
Did you know? The “Boyce” in BCNF was a relatively unsung coworker of Codd at IBM. Raymond Boyce died of a brain aneurysm at just 26 — months after co-introducing BCNF in 1974.

CHAPTER · 06
Transaction Processing Concepts

A transaction moves the DB from one consistent state to another. The bedrock guarantees are ACID.

6.1 ACID

  • Atomicity — all-or-nothing. Enforced by logging + rollback.
  • Consistency — preserves integrity constraints.
  • Isolation — concurrent execution = some serial execution.
  • Durability — committed data survives crashes. Enforced via WAL.

6.2 Transaction States

ACTIVE ─▶ PARTIALLY ─▶ COMMITTED ─▶ TERMINATED
   │      COMMITTED         ▲
   │          │             │
   ▼          ▼             │
FAILED ─────▶ ABORTED ──────┘

6.3 Schedules

  • Serial: transactions executed one after another — always correct but slow.
  • Concurrent: interleaved operations — fast but risky.
  • Serializable: equivalent to some serial schedule.
  • Recoverable: if Tj reads a value written by Ti, Ti must commit before Tj.
  • Cascadeless: Tj reads only committed values.
  • Strict: Tj neither reads nor writes a value written by an uncommitted Ti.
Strict ⊂ Cascadeless ⊂ Recoverable ⊂ All schedules

6.4 Serializability

  • Conflict serializable: precedence graph has no cycle.
  • View serializable: weaker (allows certain blind writes); test is NP-hard in general.
  • Conflict ⇒ View, but not vice-versa.
Two operations Oi(X), Oj(X) conflict iff: - they belong to different transactions, AND - at least one is a WRITE.

6.5 Building the Precedence Graph

Algorithm.
  1. One node per transaction.
  2. Add edge Ti → Tj whenever Ti has an operation conflicting with a later operation of Tj on the same data item.
  3. Schedule is conflict-serializable iff the graph is acyclic. Topological order = equivalent serial order.

6.6 Numerical Example

Schedule S: R1(A), W2(A), R3(A), W1(B), R2(B), W3(C).
Conflicts: R1(A)–W2(A) ⇒ T1→T2; W2(A)–R3(A) ⇒ T2→T3; W1(B)–R2(B) ⇒ T1→T2 (again). Edges: T1→T2, T2→T3. Acyclic ⇒ conflict serializable. Serial order: T1, T2, T3.
Pitfall: Two writes to the same item by the same transaction do not count as conflicts. Always check (a) different txns and (b) at least one write.
Did you know? The term “ACID” was coined by Andreas Reuter and Theo Härder in 1983, decades after the first transactional systems were built.

6.7 Recovery

  • Log-based: Write-Ahead Logging (WAL) — log record must hit disk before the page.
  • Deferred update (no-undo, redo).
  • Immediate update (undo, redo).
  • Checkpoints: reduce recovery work after crash.
  • Shadow paging: maintain copy-on-write page table; no logs, but expensive.
  • ARIES: industrial-grade WAL-based recovery (Analysis → Redo → Undo).

CHAPTER · 07
Concurrency Control Techniques

Concurrency control protocols ensure conflict-serializability of concurrent schedules without running everything serially. Two big families: lock-based and timestamp/optimistic.

7.1 Lock-Based Protocols

  • Shared (S) & Exclusive (X) locks.
  • 2PL (Two-Phase Locking): growing phase (acquire) + shrinking phase (release).
  • Strict 2PL: X-locks held till commit → cascadeless.
  • Rigorous 2PL: all locks held till commit → strict.
  • Conservative (Static) 2PL: acquire all locks upfront → deadlock-free, but reduces concurrency.
2PL guarantees conflict serializability, but doesn't prevent deadlocks.

7.2 Lock Compatibility Matrix

SX
S✓✗
X✗✗

7.3 Deadlock Handling

  • Prevention: ordering of resources, or timestamp-based schemes:
    • Wait-Die: older waits, younger dies.
    • Wound-Wait: older wounds (preempts) younger, younger waits.
  • Detection: Wait-for graph; cycle = deadlock; pick a victim.
  • Avoidance: Banker's algorithm (rare in DBMS).

7.4 Timestamp Ordering (TO)

  • Each Ti gets timestamp TS(Ti).
  • For each item X, track RTS(X) and WTS(X).
  • If a request violates timestamp order ⇒ abort Ti and restart with new timestamp.
On Read(X) by Ti: if TS(Ti) < WTS(X) → abort Ti else RTS(X) := max(RTS(X), TS(Ti)) On Write(X) by Ti: if TS(Ti) < RTS(X) or TS(Ti) < WTS(X) → abort Ti else WTS(X) := TS(Ti)

7.5 Multi-Version & Optimistic Protocols

  • MVCC: keep multiple versions; readers never block writers (used in PostgreSQL, Oracle).
  • OCC: read phase → validation phase → write phase. Good when conflicts are rare.

7.6 Granularity & Intention Locks

  • Locking can be at DB / file / page / row level.
  • Intention locks (IS, IX, SIX) sit on ancestors so coarse locks can detect conflicts efficiently.

7.7 Numerical — 2PL Schedule Analysis

Problem. Trace: L1-S(A), R1(A), L2-X(B), W2(B), L1-X(B) → blocked, U2(B), L1-X(B) granted, W1(B), commit2, commit1, U1.
Is this Strict 2PL? Yes — every X-lock is held till commit. Cascading rollback impossible.
Phantom problem: 2PL on existing rows doesn't prevent new rows matching a predicate from being inserted by another transaction → use predicate locks or index locking.

CHAPTER · 08
Distributed Databases

A distributed database (DDB) is a single logical DB spread over multiple physical sites connected by a network. Goals: locality, availability, scalability.

8.1 Architectures

  • Homogeneous — same DBMS everywhere.
  • Heterogeneous (federated) — different DBMSs cooperating via a global schema.
  • Shared-nothing / Shared-disk / Shared-memory — cluster topologies.

8.2 Fragmentation

  • Horizontal — partition rows (σ-based).
  • Vertical — partition columns (π-based; must keep PK).
  • Mixed/Hybrid — both.
  • Correctness rules: Completeness, Reconstruction, Disjointness (latter optional for horizontal with overlap).

8.3 Replication

  • Synchronous: all replicas updated atomically (strong consistency, higher latency).
  • Asynchronous: lazy propagation (eventual consistency).
  • Quorum (N, W, R) with W + R > N → strong consistency.

8.4 The CAP Theorem

Among Consistency, Availability, and Partition tolerance, a distributed system can guarantee at most TWO during a network partition.
In practice partition tolerance is non-negotiable for any distributed system, so the real choice is CP vs AP.

8.5 Distributed Commit — 2PC

          ┌──────────────┐
          │  COORDINATOR │
          └──────┬───────┘
   PREPARE      │
   ───────▶ all participants
                │   each: write PREPARED to log, reply YES/NO
   COMMIT/ABORT │
   ◀───────  all YES → COMMIT;   any NO → ABORT
                │   each: write COMMIT/ABORT to log, ACK
2PC blocking problem: If the coordinator crashes after participants prepared but before they hear the decision, they're stuck holding locks. 3PC adds a pre-commit phase to fix this (assumes no network partitions).

8.6 Distributed Query Processing

  • Localization of fragmented queries.
  • Semi-join: ship only the join attribute first to reduce data movement:
    R ⋉ S = πattrs(R)(R ⋈ S)
  • Cost = communication + local IO. In WAN, communication dominates.

8.7 Numerical — Semi-Join Savings

Problem. R(A,B) at site 1 has 1000 tuples × 100 B = 100 KB. S(B,C) at site 2 has 500 tuples × 80 B = 40 KB. Only 20 distinct B-values appear in S.
Naïve join: ship all of S to site 1 → 40 KB.
Semi-join: ship πB(S) → 20 × 4 B = 80 B; filter R locally; ship result back. Massive savings.

8.8 Distributed Deadlock

  • Local wait-for graphs merged into a global graph.
  • Path-pushing or edge chasing algorithms.
  • Phantom deadlocks: stale info can show false cycles.
Did you know? Google Spanner is the first commercially deployed system to claim true external consistency across a planet — thanks to GPS+atomic-clock-driven TrueTime API.

Indexing is one of the highest-yield GATE topics — expect 2–3 direct questions every year. A good index can reduce a full table scan from O(n) disk reads to O(log n).

8B.2 Dense vs Sparse Indexes

PropertyDense IndexSparse Index

Primary index is built on the ordering key of a sequentially ordered file (always sparse). Secondary index is on a non-ordering attribute (always dense, may need bucket/overflow).

8B.3 B+ Tree — The GATE Star ⭐⭐⭐⭐⭐

B+ trees are the default index structure in almost every RDBMS (InnoDB, PostgreSQL, Oracle). They give O(log n) search, insert, and delete — all guaranteed, unlike hash tables.

Order-p B+ tree: Internal node: at most p pointers, at least ⌈p/2⌉ pointers (except root) Leaf node: at most p−1 key-pointer pairs, at least ⌈(p−1)/2⌉ Height: ⌊log_⌈p/2⌉(N)⌋ where N = number of records Min records: root has 1 entry → each level fans out by ⌈p/2⌉ 
B+ Tree (Order 3, fan-out = 3):

           [ 30 | 60 ]              ← Internal node (pointers only)
          /      |     \
    [10|20]  [30|40|50]  [60|70|80]  ← Leaf nodes (data + next ptr)
       ↕         ↕           ↕
    data      data          data
    records   records       records
    ↑ Leaf nodes linked in sorted order → fast range scans ↑
Key B+ tree properties:
  • ALL data pointers are in leaf nodes only — internal nodes are just guides.
  • Leaf nodes are linked in a doubly-linked list → range queries need no backtracking.
  • The tree is always height-balanced — all leaves at the same level.
  • Splits/merges maintain the order constraint automatically.

8B.4 Numerical — B+ Tree Height & Capacity

Problem. A B+ tree index has order p = 100 (each internal node holds up to 100 pointers). The data file has N = 1,000,000 records. Find the maximum height of the tree (levels from root to leaf).

Solution.
Min fan-out per internal node = ⌈p/2⌉ = 50
Level 1 (root): at least 2 children → 2 leaf refs
Level 2: 2 × 50 = 100 leaf pointers
Level 3: 100 × 50 = 5,000
Level 4: 5,000 × 50 = 250,000
Level 5: 250,000 × 50 = 12,500,000 > 1,000,000 ✓

Answer: Max height = 4 levels (root + 2 internal + leaf). So only 4 disk reads to find any record among 1 million!
Problem 2. Internal node size = 1024 bytes. Each key = 8 bytes, each pointer = 6 bytes. What is the order p?

Solution. Node holds p pointers and (p−1) keys:
p × 6 + (p−1) × 8 ≤ 1024
6p + 8p − 8 ≤ 1024
14p ≤ 1032 → p ≤ 73.7 → p = 73

8B.7 Numerical — Index I/O Cost

Problem. A relation R has 10,000 records, block size = 512 bytes, record size = 50 bytes, pointer = 6 bytes, key = 10 bytes.
(a) Blocking factor for data = 512 / 50 = 10 records/block → 1,000 data blocks.
(b) Sparse primary index entry = key + pointer = 16 bytes → entries/block = 512/16 = 32.
Index blocks = 1,000 / 32 = ⌈31.25⌉ = 32 index blocks.
(c) Binary search on index = ⌈log₂(32)⌉ = 5 block reads + 1 data read = 6 I/Os.
vs. binary search on unsorted data = ⌈log₂(1000)⌉ = 10 I/Os — index wins!

CHAPTER · 09
Case Studies — Oracle, MySQL, PostgreSQL

Three of the most widely deployed RDBMSs in the world — each with distinct architectures, philosophies, and design trade-offs that show up frequently in interviews and GATE conceptual questions.

9.1 Oracle Database

  • Storage: data files organized into tablespaces; logical units = segments → extents → blocks.
  • Concurrency: Multi-Version Read Consistency using undo segments; readers never block writers.
  • Isolation: READ COMMITTED (default) and SERIALIZABLE; snapshot semantics under the hood.
  • Recovery: Redo logs (online + archived) + control files; ARIES-style.
  • RAC (Real Application Clusters): shared-disk active-active cluster.

9.2 MySQL (InnoDB)

  • Pluggable storage engines: InnoDB (default, transactional), MyISAM (legacy, no transactions), Memory, Archive…
  • InnoDB: clustered B+ tree on PK, supports MVCC via undo logs.
  • Replication: binlog-based (statement, row, mixed). Group replication adds Paxos-like consistency.
  • Isolation: default is REPEATABLE READ — implemented with snapshot reads + gap locks to prevent phantoms.

9.3 PostgreSQL

  • True MVCC: every row has xmin/xmax → no in-place updates; VACUUM reclaims dead tuples.
  • Index variety: B-tree, Hash, GIN, GiST, BRIN, SP-GiST.
  • Extensibility: user-defined types, operators, languages (PL/pgSQL, PL/Python…), extensions (PostGIS, pg_trgm, TimescaleDB).
  • Replication: WAL streaming; logical decoding for CDC.
  • SQL standard compliance is the highest of the three.
