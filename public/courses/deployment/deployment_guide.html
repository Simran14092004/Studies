Mastering Deployment
Ship Code with Confidence

Deployment is the process of releasing your application from a development environment to a live server where users can access it. Understanding this pipeline is critical for any developer.

Development

Your local machine. localhost:3000. Only you can see it. Changes happen here.

Staging

A mirror of production. Used for final QA testing before releasing to real users.

Production

The live application. Real users. Real data. Breaking things here has consequences.

CI/CD

Automated pipelines that build, test, and deploy your code on every push.

Dev → Staging → Production Flow

Developer pushes code
        ↓
GitHub (source of truth)
        ↓
CI runs (tests, linting, build)
        ↓ (if tests pass)
Deploy to Staging
        ↓ (QA approval)
Deploy to Production
        ↓
Monitor for errors/performance

This pipeline ensures you never push broken code to users. Each stage acts as a safety net.

CI/CD Pipeline

CI (Continuous Integration): Automatically run tests whenever code is pushed.
CD (Continuous Delivery/Deployment): Automatically deploy when tests pass.

# .github/workflows/deploy.yml
name: Deploy to Production

on:
  push:
    branches: [main]

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20
          
      - name: Install dependencies
        run: npm ci
        
      - name: Run tests
        run: npm test
        
      - name: Build
        run: npm run build
        env:
          DATABASE_URL: ${{ secrets.DATABASE_URL }}

  deploy:
    needs: build-and-test
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to Vercel
        run: npx vercel --prod --token=${{ secrets.VERCEL_TOKEN }}

Vercel — Frontend Deployment

Vercel is the easiest way to deploy React, Next.js, Vue, and static HTML apps. Zero config, global CDN, free tier.

# Method 1: CLI (Fastest)
npm i -g vercel
vercel login
vercel              # deploy to preview
vercel --prod       # deploy to production

# Method 2: Connect GitHub repo
# 1. Go to vercel.com → New Project
# 2. Import from GitHub
# 3. Configure build settings
# 4. Every push to main auto-deploys!

# vercel.json for configuration
{
  "buildCommand": "npm run build",
  "outputDirectory": "dist",
  "framework": "vite",
  "rewrites": [
    { "source": "/(.*)", "destination": "/index.html" }
  ]
}
Free Tier Limits: 100GB bandwidth/month, unlimited personal projects, automatic HTTPS, global CDN. Perfect for personal/portfolio projects.

Railway — Backend Deployment

Railway is the simplest way to deploy backend apps (Node.js, Python, Django, FastAPI) with databases included.

# Deploy Node.js/Express app
# 1. Push code to GitHub with a start script in package.json:
{
  "scripts": {
    "start": "node server.js"
  }
}

# 2. Connect Railway to GitHub repo
# 3. Add environment variables in Railway dashboard
# 4. Add a database (PostgreSQL, MySQL, Redis) with one click

# For Python/FastAPI:
# Procfile (create in root)
web: uvicorn main:app --host 0.0.0.0 --port $PORT

# requirements.txt must exist
pip freeze > requirements.txt

Docker — Containerization Basics

Docker packages your app and all its dependencies into a container that runs identically everywhere — no more "works on my machine"!

# Dockerfile for a Python Flask app
FROM python:3.11-slim

WORKDIR /app

# Copy and install dependencies first (caching optimization)
COPY requirements.txt .
RUN pip install -r requirements.txt

# Copy application code
COPY . .

EXPOSE 5000
CMD ["python", "app.py"]

# Dockerfile for Node.js
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]
# Common Docker commands
docker build -t my-app .          # Build image
docker run -p 3000:3000 my-app    # Run container
docker ps                          # List running containers
docker stop container_id           # Stop container
docker-compose up                  # Start multi-service setup
docker-compose down                # Stop all services

# docker-compose.yml (full stack example)
version: '3.8'
services:
  web:
    build: .
    ports: ["3000:3000"]
    env_file: .env
    depends_on: [db]
  db:
    image: postgres:15
    environment:
      POSTGRES_PASSWORD: secret

VPS Deployment (Linux Server)

Virtual Private Servers (DigitalOcean, Linode, AWS EC2) give you full control. More work, but maximum flexibility.

# 1. SSH into your server
ssh root@your-server-ip

# 2. Install Node.js
curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
sudo apt install nodejs

# 3. Clone your project
git clone git@github.com:yourname/project.git
cd project && npm install

# 4. Install PM2 (keeps app running)
npm i -g pm2
pm2 start server.js --name "my-app"
pm2 startup          # auto-start on server reboot
pm2 save

# 5. Check status
pm2 status
pm2 logs my-app

Nginx & Reverse Proxy

Nginx sits in front of your app, handling HTTP/HTTPS traffic and routing to your Node/Python process.

# Install Nginx
sudo apt install nginx

# Configure site (/etc/nginx/sites-available/myapp)
server {
    listen 80;
    server_name yourdomain.com www.yourdomain.com;

    location / {
        proxy_pass http://localhost:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_cache_bypass $http_upgrade;
    }
}

# Enable the site
sudo ln -s /etc/nginx/sites-available/myapp /etc/nginx/sites-enabled/
sudo nginx -t          # test config
sudo systemctl reload nginx

# Free SSL with Let's Encrypt
sudo apt install certbot python3-certbot-nginx
sudo certbot --nginx -d yourdomain.com

Environment Variables & Secrets

NEVER commit secrets to Git! No API keys, passwords, or database URLs in your code. Use .env files locally and platform secrets in production.
# .env (local development — add to .gitignore!)
DATABASE_URL=postgresql://localhost:5432/mydb
JWT_SECRET=super-secret-key-here
OPENAI_API_KEY=sk-...
PORT=3000

# .gitignore — ALWAYS include:
.env
.env.local
.env.production

# In Node.js (using dotenv)
npm install dotenv
require('dotenv').config()
const dbUrl = process.env.DATABASE_URL

# In Python
from dotenv import load_dotenv
import os
load_dotenv()
db_url = os.getenv("DATABASE_URL")

In Vercel: Settings → Environment Variables. In Railway: Variables tab. Never hardcode!

Pre-Deploy Checklist

✅ All tests passing
✅ Environment variables set in production
✅ Database migrations run
✅ .env not in git
✅ Build succeeds locally first
✅ Error handling & logging in place
✅ HTTPS/SSL configured
✅ Performance: images optimized, minified CSS/JS
✅ Security headers set
✅ Backup plan if deployment fails (rollback strategy) 
# Quick rollback with Vercel
vercel rollback    # reverts to previous deployment

# With Git (tag before deploying)
git tag v1.2.0 && git push origin v1.2.0
# If something breaks:
git revert HEAD
git push origin main
