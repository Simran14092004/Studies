CH 01

Introduction & Basics

What is an Operating System?

An OS is a resource manager and hardware abstraction layer — it manages CPU, memory, I/O devices and provides a clean interface so user programs need not deal with hardware directly.

Key roles: Resource manager · Process scheduler · Memory manager · File system manager · Security enforcer · User interface provider

OS Structure Types

TypeDescriptionExampleGATE
MonolithicAll OS services in single large kernel. Fast but hard to maintain.Linux, Unix
MicrokernelMinimal kernel; services run in user space. More stable, slower IPC.Mach, QNX⭐
LayeredOS divided into layers (hardware → user). Each layer uses layer below.THE system
Virtual MachineProvides illusion of multiple OS instances over single hardware.VMware, JVM
ExokernelExposes raw hardware to apps. Minimal abstraction.Research

System Calls

The interface between a user process and the OS kernel. User programs trap into kernel mode via system calls. Categories:

  • Process control: fork(), exec(), exit(), wait()
  • File management: open(), read(), write(), close()
  • Device management: ioctl(), read(), write()
  • Memory management: mmap(), brk()
  • Communication: pipe(), socket()

CH 04

Synchronisation & Deadlock

Critical Section Requirements

A correct critical-section solution must satisfy all three:
1. Mutual Exclusion — At most one process in CS at a time.
2. Progress — If no one is in CS and some want to enter, one must eventually enter.
3. Bounded Waiting — A process must enter CS within finite time after requesting.

Semaphores

// Binary semaphore (mutex): value ∈ {0,1}
wait(S):   if (S > 0) S-- else block process
signal(S): S++; if (blocked) wake one up

// Counting semaphore: for managing n resources
// Classic problems:
// Producer-Consumer → sem full=0, empty=N, mutex=1
// Readers-Writers   → sem rw_mutex=1, mutex=1
// Dining Philosophers → tricky! (asymmetry solution)
Spinlock vs Block: Spinlocks waste CPU (busy-wait) but are fast for very short waits. Blocking semaphores use OS sleep/wake — better for longer waits.

Peterson's Solution ⭐⭐⭐⭐ GATE Favourite

🎯 GATE Loves This
Peterson's solution is a software-only mutual exclusion algorithm for 2 processes. GATE frequently asks which conditions it satisfies and how the turn/flag variables work.
// Peterson's Solution for 2 processes (Pi and Pj)
boolean flag[2] = {false, false};  // flag[i] = "I want to enter CS"
int turn;                           // turn = whose turn it is

// Code for process Pi:
do {
    flag[i] = true;          // I want to enter
    turn = j;                // Give turn to other process (generous!)
    while (flag[j] && turn == j)
        ;   // Busy wait — only blocked if OTHER wants in AND it's their turn

    // CRITICAL SECTION

    flag[i] = false;         // I'm done
    // REMAINDER SECTION
} while (true);
PropertySatisfied?Why
Mutual Exclusion✓ YesIf both in CS, turn = i AND turn = j simultaneously — impossible
Progress✓ YesIf one doesn't want to enter (flag=false), the other proceeds immediately
Bounded Waiting✓ YesAfter Pi sets turn=j, Pi enters CS before Pj gets a second chance
Busy Waiting✗ Yes (bad)Wastes CPU cycles in the while loop — not ideal for long waits

Classical Synchronization Problems ⭐⭐⭐⭐

1. Producer-Consumer (Bounded Buffer)

// Semaphores:
semaphore mutex = 1;    // mutual exclusion on buffer
semaphore full  = 0;    // count of full slots
semaphore empty = N;    // count of empty slots (N = buffer size)

Producer:                      Consumer:
do {                           do {
  produce item;                  wait(full);
  wait(empty);                   wait(mutex);
  wait(mutex);                   remove item from buffer;
  add item to buffer;            signal(mutex);
  signal(mutex);                 signal(empty);
  signal(full);                  consume item;
} while(true);               } while(true);
Key insight: The order of wait() calls matters. wait(mutex) must come AFTER wait(full) / wait(empty), otherwise deadlock can occur if buffer is full/empty.

2. Readers-Writers Problem

// Semaphores:
semaphore rw_mutex = 1;  // exclusive access for writers
semaphore mutex    = 1;  // protects read_count
int read_count     = 0;

Writer:                        Reader:
wait(rw_mutex);                wait(mutex);
  ... write ...                  read_count++;
signal(rw_mutex);                if (read_count == 1)
                                   wait(rw_mutex);  // first reader blocks writers
                               signal(mutex);
                               ... read ...
                               wait(mutex);
                                 read_count--;
                                 if (read_count == 0)
                                   signal(rw_mutex); // last reader releases
                               signal(mutex);
Issue: This solution favours readers. Writers can starve if readers keep arriving. Second Readers-Writers problem inverts this — writers get priority.

3. Dining Philosophers Problem

5 philosophers sit at a round table. Each needs 2 forks (left + right) to eat. Naive solution deadlocks if all pick up left fork simultaneously.

// Asymmetric Solution (avoids deadlock):
// Even philosopher picks LEFT then RIGHT
// Odd philosopher picks RIGHT then LEFT
semaphore fork[5] = {1,1,1,1,1};
philosopher(i):
  if (i % 2 == 0):
    wait(fork[i]);         // left
    wait(fork[(i+1)%5]);   // right
  else:
    wait(fork[(i+1)%5]);   // right first
    wait(fork[i]);         // left
  ... eat ...
  signal(fork[i]);
  signal(fork[(i+1)%5]);
Solution StrategyHowDeadlock-free?
Allow max 4 philosophers to sitSemaphore initialized to 4Yes
Asymmetric pickup (even/odd)Even: L then R; Odd: R then LYes
Pick both or noneUse mutex to check both availableYes

Demand Paging Internals ⭐⭐⭐⭐ GATE Essential

In demand paging, pages are loaded into memory only when referenced. Each page table entry contains control bits:

BitNameValue 0Value 1Use
VValid/Invalid bitPage not in memory → Page FaultPage in memory (valid)Set to 0 at load, 1 when loaded into frame
DDirty/Modified bitPage unchanged since loadPage was written toIf D=0, eviction is free (no write-back). If D=1, must write to disk first.
RReference/Access bitNot recently usedRecently accessedUsed by Clock algorithm for LRU approximation; OS periodically clears it
Page Table Entry: | Frame Number | V | D | R | Protection bits |
🎯 GATE Tip
When a page fault occurs: (1) Trap to OS, (2) Find a free frame, (3) Load page from disk, (4) Update page table (set valid bit = 1), (5) Restart the faulting instruction. If no free frame exists, a page replacement algorithm selects a victim (check dirty bit to decide if write-back is needed).

Belady's Anomaly — Explained ⭐⭐⭐ GATE Favourite

⚠️ COUNTER-INTUITIVE
Belady's Anomaly: For FIFO page replacement, adding MORE frames can actually cause MORE page faults. This violates intuition — more resources should always help, but FIFO doesn't have the stack property.

Classic example: Reference string: 1 2 3 4 1 2 5 1 2 3 4 5

FramesAlgorithmPage FaultsAnomaly?
3FIFO9—
4FIFO10 ← MORE!✓ Anomaly occurs!
3LRU8No anomaly
4LRU6 ← Fewer (correct)No anomaly
Why LRU/OPT are immune: They are stack algorithms — the set of pages in memory with n frames is always a subset of the pages with n+1 frames. This stack property guarantees that adding frames never hurts. FIFO lacks this property.

Thrashing — Deep Dive ⭐⭐⭐⭐

FIG 6.1 — CPU utilisation vs. degree of multiprogramming
ConceptDefinitionGATE angle
ThrashingProcess spends more time in paging (page-in/page-out) than actual CPU executionCaused by too many processes competing for limited frames
Locality of ReferencePrograms tend to access a small set of pages repeatedly in any phase of executionFoundation of all page replacement algorithms; makes caching effective
Working SetSet of pages a process actively uses in a time window Δ (e.g. last 10 references)Working set size = frames process needs; if total WSS > available frames → thrashing
Page Fault FrequencyRate at which page faults occur for a processIf PFF too high → give more frames. If too low → take frames away. Control loop for thrashing prevention.
🎯 Prevention Strategies
1. Working Set Model: Track each process's working set; only allow multiprogramming if Σ(WSS) ≤ available frames.
2. Page Fault Frequency Control: If PFF exceeds upper threshold → add frames. If below lower threshold → remove frames.
3. Reduce Multiprogramming: Suspend processes and free their frames for remaining processes.

Stack algorithms (LRU, OPT) have the property that the set of pages in memory with n frames is always a subset of the set with n+1 frames. This guarantees no Belady's anomaly. FIFO is NOT a stack algorithm.

GATE 2021
In C-SCAN disk scheduling, after reaching the highest numbered cylinder in the queue, the head:
AReverses direction and services requests back
BImmediately jumps to the lowest-numbered cylinder and services upward again
CStops and waits for new requests
DGoes to cylinder 0 always (not just lowest request)
C-SCAN (Circular SCAN) wraps around — after reaching the highest request cylinder, it immediately jumps to the lowest-numbered requested cylinder (or 0 in some variants) and services upward again. This gives more uniform wait times than SCAN.

🎯 GATE Tip
A zombie consumes a process table entry but no CPU or significant memory. Too many zombies can exhaust the process table. A zombie is NOT the same as an orphan.

fork() + exec() Relationship

// fork() creates a copy of the current process
pid_t pid = fork();
if (pid == 0) {
    // Child process — runs concurrently
    exec("new_program");  // Replace child's image with new program
} else {
    // Parent process — pid = child's PID
    wait(NULL);           // Wait for child to finish
}
CallWhat it doesReturns
fork()Creates child process — exact copy of parent (code, data, heap, stack). Both resume after fork().Parent: child PID | Child: 0 | Error: -1
exec()Replaces current process image with a new program. Does NOT create a new process — same PID.Never returns on success (new program takes over)
Copy-on-Write (COW)fork() doesn't actually copy pages — marks them read-only/shared. Page is copied only when first written.Huge performance win — exec() after fork() wastes no copy effort

Linux Completely Fair Scheduler (CFS)

Linux (kernel 2.6.23+) replaced the traditional O(1) scheduler with CFS — a fair-share scheduler based on virtual runtime.

ConceptExplanation
Virtual Runtime (vruntime)Tracks how much CPU time each process has used (weighted by priority). CFS always picks the process with the smallest vruntime.
Red-Black TreeRunnable processes sorted by vruntime in a self-balancing BST. O(log n) scheduling decisions.
Nice valueAdjusts weight: lower nice = more CPU (default nice=0). Range: -20 (highest priority) to +19 (lowest).
No fixed quantumTarget latency divided among runnable processes. Each gets time proportional to its weight.
PreemptionOn wake-up, if new process has much smaller vruntime than running process, preemption occurs immediately.
vs Traditional: Traditional priority scheduling can starve low-priority processes. CFS guarantees each process gets a fair share of CPU proportional to its weight — no starvation possible.

Extra · Paging AddressLogical to Physical Address Translation

Page size = 512 bytes. Page table: P0→Frame 3, P1→Frame 7, P2→Frame 2. Logical address = 1026. Find physical address.

▶ Show solution
1Offset bits = log₂(512) = 9 bits. Page# bits = remaining.
2Logical 1026 in binary: 1026 = 10000000010₂ | Page# = 1026 / 512 = 2 (integer division). Offset = 1026 mod 512 = 2.
3Page 2 → Frame 2 (from page table).
4Physical = Frame × Page_size + Offset = 2 × 512 + 2 = 1026
✓ Physical Address = 1026 (coincidence here — frame equals page number)

Extra · PagingTwo-Level Page Table Memory

32-bit address, page size = 4 KB, two-level paging: outer PT 10 bits, inner PT 10 bits, offset 12 bits. How many entries in outer page table? How much memory for a full outer PT (each entry = 4 bytes)?

▶ Show solution
1Outer PT bits = 10 → entries = 2¹⁰ = 1024 entries
2Memory for outer PT = 1024 × 4 bytes = 4 KB (fits in one page!)
3Each inner PT also has 2¹⁰ = 1024 entries × 4 bytes = 4 KB. Max inner PTs = 1024 → total PT space = 4 MB + 4 KB.
✓ Outer PT: 1024 entries, 4 KB. Full 2-level PT ≈ 4 MB
